# Browser OAuth Flow

How a user signs in through a browser-based MCP client (Inspector, Claude Desktop, mcpc, custom React app). Same flow for DCR and Proxy modes; the differences are noted inline.

## High-level steps

1. Client fetches `GET /.well-known/oauth-authorization-server` from the MCP server.
2. Client picks endpoints from the metadata.
3. Client generates PKCE verifier + challenge (`code_challenge_method=S256`).
4. **DCR:** client posts to upstream `registration_endpoint` and receives a `client_id`.
   **Proxy:** client posts to MCP server's `/register` and receives the server's pre-configured `client_id`.
5. Client opens browser to authorize URL with `redirect_uri`, `state`, `code_challenge`, `code_challenge_method=S256`, `scope`.
6. User signs in upstream and consents. Browser redirects to `redirect_uri` with `?code=...&state=...`.
7. Client posts to token endpoint with `code`, `code_verifier`, `client_id`.
   **Proxy:** server intercepts, injects `client_secret`, forwards to upstream.
8. Token endpoint returns `access_token` (+ optional `refresh_token`).
9. Client uses `Authorization: Bearer <access_token>` on every `/mcp/*` request.

## Redirect URIs

The redirect URI is registered with the upstream provider (or with the proxy server). Common shapes:

| Client | Redirect URI |
|---|---|
| MCP Inspector | `http://localhost:<port>/inspector/oauth/callback` |
| Claude Desktop | `http://localhost:<dynamic>/oauth/callback` |
| mcpc | `http://localhost:<dynamic>/oauth/callback` |
| Browser app | `https://app.example.com/oauth/callback` |

Inspector and mcpc bind to dynamic ports per session. Configure providers to allow `http://localhost:*/**` in dev (Supabase, Auth0, WorkOS dashboards all support wildcards). For production, use exact HTTPS URIs.

## State and PKCE

- **`state`** — opaque random string, validated on the redirect to prevent CSRF. Generated by the client.
- **`code_verifier`** — random 43-128 char string; SHA-256 hashed and base64url-encoded into `code_challenge`. Required by OAuth 2.1.
- **`code_challenge_method=S256`** — only `S256` is accepted. Plain is forbidden.

mcp-use enforces PKCE for both DCR and Proxy modes. Clients without PKCE are rejected.

## Inspector "OAuth Mode" toggle

MCP Inspector ships with two modes:

| Mode | What it does |
|---|---|
| **Auth disabled** | Sends no `Authorization` header. Server returns 401 if OAuth is configured. Useful only when the server has no `oauth` option. |
| **OAuth Mode** | Performs the full discovery + DCR/proxy + PKCE + token flow. Stores tokens in browser storage. |

Toggle is in the Inspector header. Select **OAuth Mode** for any server with `oauth` configured. The Inspector logs every step in its console — see `../20-inspector/` for setup.

## Common pitfalls

| Pitfall | Symptom | Fix |
|---|---|---|
| Redirect URI not allowed upstream | `redirect_uri_mismatch` after consent | Add Inspector/mcpc URI to provider dashboard, including wildcard for dev ports |
| `state` not preserved | Inspector errors after redirect | Bug in client; not a server issue |
| PKCE plain instead of S256 | Token endpoint rejects | Update client to S256 |
| Server runs HTTP, provider requires HTTPS redirect | Provider refuses redirect | Use a tunnel (cloudflared, ngrok) — see `../21-tunneling/` |
| `.well-known` returns CORS error | Browser blocks discovery | Add CORS headers — see `../26-anti-patterns/05-security-and-cors.md` |
| User already signed in upstream, no consent shown | Tokens have stale scopes | Force re-consent with `extraAuthorizeParams: { prompt: 'consent' }` (proxy only) |

## Tunneling for HTTPS redirects

Some providers reject `http://` redirect URIs even for localhost (Slack, some Azure AD configs). Run a tunnel and configure the upstream redirect to the tunnel URL:

```bash
cloudflared tunnel --url http://localhost:3000
# Use the resulting https://*.trycloudflare.com URL as redirect_uri
```

See `../21-tunneling/` for the canonical setup.
