# Layer Provide
Use `Layer.provide` to feed provider outputs into a dependent layer while hiding the provider outputs from the final layer.

## Core Semantics

`Layer.provide(inner, outer)` means: use `outer` to satisfy requirements of `inner`.

```typescript
import { Context, Effect, Layer } from "effect"

class ConfigService extends Context.Tag("app/ConfigService")<
  ConfigService,
  { readonly apiUrl: string }
>() {}

class UsersClient extends Context.Tag("app/UsersClient")<
  UsersClient,
  { readonly findName: (id: string) => Effect.Effect<string> }
>() {}

const ConfigLive = Layer.succeed(ConfigService, {
  apiUrl: "https://example.test"
})

const UsersClientLive = Layer.effect(
  UsersClient,
  Effect.gen(function* () {
    const config = yield* ConfigService
    return {
      findName: (id: string) => Effect.succeed(`${config.apiUrl}/users/${id}`)
    }
  })
)

const UsersClientReady = UsersClientLive.pipe(
  Layer.provide(ConfigLive)
)
```

`UsersClientReady` produces `UsersClient`; it does not also produce `ConfigService`.

## Type Shape

The v3 source describes `provide` as feeding one builder's outputs into another builder's inputs. The simplified type transition is:

```typescript
Layer.Layer<UsersClient, never, ConfigService>
Layer.Layer<ConfigService, never, never>
Layer.Layer<UsersClient, never, never>
```

The provider's output satisfied the requirement and then disappeared from the output set.

## When Hiding Is Correct

Hiding provider outputs is right when the provider is purely an implementation detail.

| Scenario | Why `provide` fits |
|---|---|
| Database needs config | Consumers should use database, not config |
| HTTP client needs TLS settings | TLS settings are internal |
| Repository needs connection pool | Program should not depend on the pool |
| Service default owns dependencies | `.Default` generated by `Effect.Service` uses this shape |

## Pipe Direction

In pipe style, the dependent layer is on the left.

```typescript
const Ready = DependentLive.pipe(
  Layer.provide(DependencyLive)
)
```

Read it as: "Build `DependentLive` by providing `DependencyLive`."

## Array Form

The v3 source includes overloads for providing a non-empty array of layers.

```typescript
const UsersClientReady2 = UsersClientLive.pipe(
  Layer.provide([ConfigLive])
)
```

For many layers, `Layer.mergeAll` often reads better because it names independent composition explicitly before providing it.

```typescript
const BaseLive = Layer.mergeAll(ConfigLive, LoggerLive)
const Ready = UsersClientLive.pipe(Layer.provide(BaseLive))
```

## Main Gotcha

If the final program needs both the dependent service and the provider service, `Layer.provide` is the wrong combinator because it hides the provider output.

```typescript
const TestLive = UsersClientLive.pipe(
  Layer.provide(ConfigLive)
)
```

`TestLive` cannot satisfy a program that yields both `UsersClient` and `ConfigService`. Use `Layer.provideMerge` for that.

## Nested Provide Chains

Deep `provide` chains are correct when every dependency is internal, but they can become hard to read.

```typescript
const HttpReady = HttpClientLive.pipe(
  Layer.provide(ConfigLive)
)

const UsersReady2 = UsersClientLive.pipe(
  Layer.provide(HttpReady)
)
```

Name intermediate layers when the graph has more than one conceptual step. The types are easier to inspect and the hidden outputs are intentional.

## App Root Rule

Before using `Layer.provide` at the app root, ask whether the provider service should be visible to the program.

| Provider service | Usually hidden? |
|---|---:|
| Low-level socket pool | Yes |
| Internal config parser | Yes |
| App config service | Maybe |
| Test helper service | Usually no |

When unsure in tests, prefer `Layer.provideMerge` and narrow later.

## Generated Defaults

`Effect.Service` with `dependencies` uses this hidden-provider model for `.Default`. That is convenient for application defaults and occasionally surprising in tests.

## Type Reading Example

```typescript
declare const Inner: Layer.Layer<"Inner", never, "Dep">
declare const Dep: Layer.Layer<"Dep", never, never>

const Ready = Inner.pipe(Layer.provide(Dep))
```

`Ready` produces only `"Inner"`. If the program still yields `"Dep"`, the final effect still has a missing requirement.

That hidden output is the point of `Layer.provide`.
